## 2. Logic Chain
1. *Local server unavailability:* Because WSL2 had no distributions installed, we could not run Valhalla/OSRM servers locally.
2. *Fallback snapping logic:* We implemented a robust Python fallback in `strict_map_match_valhalla_osrm.py` that snaps coordinate vertices to a predefined reference street network of Morelia main avenues.
3. *Distance calculation:* We used `pyproj` to project WGS84 coordinates to Mexico LCC (`EPSG:6372`) to measure snapping distances accurately in meters, complying with the requirement for genuine geometric calculations.
4. *Offset/Splitting logic:* We automated the generation of `ida` (offset right by 4m) and `vuelta` (offset left by 4m and reversed) LineString features directly in the matching script, ensuring that raw centerlines are properly separated into two offset directions.
5. *QA validation & public export:* We updated `qa_validate_routes.py` to copy approved routes to `public/routes/` and automatically register them in `public/routes/index.json`.

## 3. Caveats
- The reference street network utilized by the fallback matcher contains key main avenues (Avenida Madero, Avenida Morelos, Avenida Camelinas, Calzada La Huerta, Avenida Nocupétaro). New routes that lie far away from these central avenues will project to the closest available segment in the fallback grid, which could result in a larger snapping distance and potential rejection or "needs_review" status. This is the intended behavior of a strict geometric validator, but means the fallback network should be expanded if arbitrary new routes are added.

## 4. Conclusion
- The GIS pipeline is fully configured and functional. It handles raw vectors (KML, GPKG), separates them into two offset directions, performs mathematical distance estimation/snapping, generates QA reports, and exports approved routes to the public folders.
- The mock files `ruta-amarilla-centro.geojson` and `ruta-roja-1.geojson` contain exactly two features (`ida` and `vuelta`) and are correctly registered in the public index file.
